# Amazon Review Crawler

Amazon 제품 리뷰를 자동으로 수집하고, 텍스트 분석을 통해 핵심 키워드를 추출한 뒤 재검색하는 2단계 크롤링 파이프라인입니다.


## 1. 프로젝트 개요

Amazon은 기본적으로 단순 스크롤 방식으로는 최대 100개의 리뷰만 조회할 수 있습니다. 본 크롤러는 이 제한을 **키워드 검색 기반 2단계 크롤링**으로 우회하며, 수집된 리뷰를 텍스트 분석하여 추가 키워드를 자동으로 생성·재검색하는 구조를 갖고 있습니다.

---

## 2. 주요 기능

| 기능 | 설명 |
|---|---|
| **2단계 크롤링** | 초기 수집 → 키워드 분석 → 키워드 기반 재수집 |
| **키워드 자동 추출** | CountVectorizer + TF-IDF 결합 분석 |
| **중복 리뷰 제거** | 작성자 + 리뷰 내용 기준 자동 deduplication |
| **구매 옵션 파싱** | 옵션 문자열(`key: value \| key: value`)을 개별 컬럼으로 분리 |
| **불용어 처리** | 제품명, 사용자 지정 단어 제거 |
| **사용자 지정 키워드** | 재구매·리뷰 업데이트 관련 고정 검색어 수동 추가 가능 |

---

## 3. 동작 방식

```
[STEP 1] 초기 크롤링
    └─ INITIAL_URL 접속 → 최초 N 페이지 리뷰 수집

[STEP 2] 키워드 추출
    └─ 수집된 리뷰 텍스트에 CountVectorizer + TF-IDF 적용
    └─ 불용어(제품명, MY_STOPWORDS) 제거 후 상위 키워드 선정
    └─ refined_wordlist()로 키워드 정제 (접미사 중복 제거)
    └─ SELF_SEARCH_WORDS(사용자 지정 키워드) 병합

[STEP 3] 키워드 기반 재크롤링
    └─ 각 키워드를 검색창에 입력 → 해당 키워드가 포함된 리뷰 수집

[STEP 4] 데이터 병합 및 저장
    └─ 초기 데이터 + 키워드별 데이터 병합
    └─ 중복 제거 → 옵션 컬럼 파싱 → CSV 저장
```

## 4. 설정 및 실행

### 4.1 주요 설정값 (`Cell 10`)

```python
PRODUCT_NAME = 'The Heaven & Earth Grocery Store - 2023'   # 제품명 (불용어 처리에 사용)
INITIAL_URL  = 'https://www.amazon.com/product-reviews/...' # 전체 리뷰 페이지 URL

INITIAL_MAX_PAGES  = 10   # 초기 수집 페이지 수
KEYWORD_MAX_PAGES  = 10   # 키워드별 재수집 페이지 수
TOP_N_KEYWORDS     = 10   # 추출할 상위 키워드 개수

MY_STOPWORDS       = ['don', 'like', 've']      # 사용자 정의 불용어
SELF_SEARCH_WORDS  = ['conflict']               # 고정 추가 검색어
```

### 4.2 실행 순서

1. ChromeDriver 및 의존 패키지 설치
2. `PRODUCT_NAME`과 `INITIAL_URL` 설정
3. 코드 실행 후 브라우저에서 **CAPTCHA 수동 해결 및 로그인 문제 해결**
4. 리뷰 페이지가 로드되면 터미널에서 `Enter`
5. 자동 수집 진행 → 결과 CSV 파일 저장

### 4.3 출력 파일

```
amazon_reviews_{PRODUCT_NAME}_final.csv
```

## 5. 알려진 문제 및 해결 방안

### 5.1 🚨 봇 감지 및 계정 차단 위험

#### 문제

Selenium을 이용한 자동화 크롤링은 Amazon의 봇 감지 시스템에 의해 차단될 수 있습니다.  
현재 코드는 다음과 같은 **부분적 우회 수단**을 포함하고 있습니다.

- `human_delay()` : 페이지 이동 간 랜덤 대기 (2~9초)
- `human_scroll()` : 랜덤 픽셀 단위의 자연스러운 스크롤 시뮬레이션

#### 현재 권장 방안

> **계정당 1회 일회성 시도** 방식이 현실적인 대안입니다.  
> 단일 계정으로 대량 반복 요청 시 차단 위험이 높아지므로, 여러 계정을 로테이션하거나 실행 빈도를 제한하는 것을 권장합니다.

#### 추가 고려 가능한 방법 (미구현)

- User-Agent 랜덤 변경
- Residential Proxy 적용
- 요청 간격 더욱 확대 (키워드 사이 10~30초 대기)
- CAPTCHA 해결 후 세션 쿠키 재사용

### 5.2 ⚠️ 키워드 중복 문제 (어형 변화 미처리)

#### 문제

현재 키워드 추출 과정에서 **어형 변화형(inflected form)이 별도 키워드로 중복 추출**되는 현상이 발생합니다.

**예시**

| 추출된 단어 | 원형 |
|---|---|
| `love` | love |
| `loved` | love |
| `loving` | love |

결과적으로 동일한 의미를 가진 단어들이 별개의 검색 쿼리로 실행되어 **불필요한 중복 크롤링**이 발생합니다.

#### 원인

현재 `refined_wordlist()` 함수는 접미사(`s`, `es`, `ed`, `ing`)를 단순 제거하는 방식으로 동작하며, 이는 **불완전한 어간 추출(naïve stemming)** 입니다.  
`loved` → `lov` (잘못된 어간), `love` → 그대로 유지되는 경우처럼 두 단어가 같은 어근임을 인식하지 못합니다.

#### 해결 방안 (예정)

키워드 확정 전에 **서브워드(subword) 기반 재처리 단계**를 추가할 예정입니다.
